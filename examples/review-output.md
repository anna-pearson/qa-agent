# Example: Review Output

> This is a real review generated by `qa-agent review` against a [DJ Mix Player test suite](https://github.com/anna-pearson/playwright-test-suite) with 167 tests. The agent read the source code, ran the tests, and produced this report automatically.

## Playwright Test Suite Review — MixDeck DJ Player

### 1. Summary

All 167 tests pass across three well-structured files. The suite is above average in terms of organisation: a solid Page Object Model (POM) is used throughout `dj-player.spec.ts`, a proper `beforeEach` reset guard is in place in `api.spec.ts`, and the audio stub is clever and functional. That said, there are real issues lurking below the surface — several tests have logic bugs that mask actual behaviour rather than verify it, the audio stub has a shared-state race condition that will bite on parallel runs, a handful of accessibility tests degenerate into presence checks rather than role checks, and several important behaviours of the app are entirely untested.

---

### 2. Critical Issues

#### Audio stub uses shared prototype-level state — parallel tests will corrupt each other
**File:** `tests/dj-player.spec.ts`, lines 12–67 (`stubAudio`)

The stub patches `HTMLAudioElement.prototype` with **module-level variables** (`_currentTime`, `_interval`, `_lastInstance`, etc.) shared across all instances. Because Playwright runs multiple workers and each worker can run multiple tests in the same browser context (or the page can create multiple `Audio` objects), a second `new Audio()` call resets `_currentTime` to 0 and clears the `_interval` of the *previous* audio object.

**Fix:** Move all state inside a per-instance closure using an ES6 `class` approach or a `WeakMap` keyed on `this`, rather than closed-over module variables on the prototype.

---

#### `prevTrack` wrap-around test is invalid — the stub defeats the "restart if > 3s" guard
**File:** `tests/dj-player.spec.ts`, lines 499–507

The test passes for the right reason, but only by coincidence — `_currentTime` happens to be under 3 seconds when the Prev button is clicked. If the stub were faster or the test machine slower, the test would fail. There is no explicit control of `currentTime` before calling prev.

**Fix:** Explicitly set `audio.currentTime = 0` via evaluate before clicking Prev to make the precondition deterministic.

---

#### "Whitespace search" edge-case test makes a vacuous assertion

```ts
const count = await player.trackItems.count();
expect(count).toBeGreaterThanOrEqual(0);
```

`count >= 0` is always true. A regression where whitespace search returns 0 tracks (broken) would silently pass.

**Fix:** Assert `await expect(player.trackItems).toHaveCount(0)` (documenting current behaviour).

---

#### Seeking tests don't assert the exact time — weak verification

```ts
await expect(player.timeCurrent).not.toHaveText('0:00');
```

This only proves *something* happened. An off-by-one (seeking by 1s instead of 5s) would pass. The ArrowLeft test uses `textContent()` instead of a web-first assertion — a latent flakiness risk.

**Fix:** Assert the exact expected time with `toHaveText('0:05')`.

---

#### Several Accessibility tests only check visibility, not the semantic role/label

Six tests use `locator('section[aria-label="..."]')` + `toBeVisible()`. This implicitly checks the aria-label exists but does not verify the element maps to `role="region"` or that a screen reader would announce it as a landmark.

**Fix:** Use `page.getByRole('region', { name: 'Now playing' })` which explicitly requires both the ARIA role AND the accessible name.

### 3. Coverage Gaps

| Gap | Description |
|-----|-------------|
| `prevTrack` restart | When Prev is clicked and `audio.currentTime > 3`, it rewinds instead of going back — completely untested |
| Mouse-drag seeking | Progress bar supports click/drag to seek — no test covers this |
| EQ bars animation | Animated EQ bars appear on active track while playing — untested |
| Network failure handling | No test for what happens if `/api/tracks` returns 500 or times out |
| API field validation | No tests for empty string title, non-numeric BPM, or missing optional fields |
| `freq`/`color` in schema | API schema test checks 7 fields but server returns 9 — two fields could be removed silently |
| ID counter after delete | After deleting a track, the next POST should get the next sequential ID — never tested |
| `loadedmetadata` path | Duration display test can't distinguish synchronous vs async update path |
| Tab-order navigation | Tracklist items have `tabindex="0"` but Tab-key focus order is untested |
| `/api/tracks/reset` | The reset endpoint is called in `beforeEach` but its own behaviour is never directly tested |
